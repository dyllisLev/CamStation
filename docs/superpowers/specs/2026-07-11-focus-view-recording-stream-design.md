# 집중보기 고화질 스트림 설계

## 목표

`/live`의 일반 격자는 기존 라이브 스트림을 계속 사용하고, `집중 보기`로 확대한 타일만 녹화용 고화질 스트림을 재생한다. 기존 ffmpeg 녹화 작업과 녹화 설정은 변경하지 않는다.

## 현재 구조

- 카메라 API는 안정적인 카메라 키인 `streamName`과 역할별 `liveStreamName`, `recordingStreamName`을 제공한다.
- 일반 라이브 타일은 `liveStreamName`을 우선해 go2rtc MSE에 연결한다.
- 집중보기는 격자 위에 별도의 타일을 렌더링하므로 진입할 때 새 MSE 연결을 만들고, 종료할 때 그 연결을 정리한다.
- 녹화기는 브라우저 연결과 별개로 `recordingStreamName`에 해당하는 로컬 go2rtc RTSP 입력을 계속 사용한다.

## 설계

재생 스트림 선택 규칙만 집중보기 여부에 따라 나눈다.

- 일반 타일: `liveStreamName` → `streamName`
- 집중보기 타일: `recordingStreamName` → `liveStreamName` → `streamName`

집중보기 타일에만 집중보기 상태를 전달하고, 기존 MSE 훅에는 선택된 스트림 이름을 그대로 넘긴다. 백엔드 API, go2rtc 설정 생성, 녹화기 관리 코드는 변경하지 않는다.

## 동작과 예외 처리

- 집중보기 진입 시 고화질 스트림을 새로 연결하므로 기존과 마찬가지로 잠시 `연결 중...`이 표시될 수 있다.
- 집중보기를 닫으면 고화질 MSE 연결은 정리되고 일반 격자의 저화질 라이브 연결은 유지된다.
- `recordingStreamName`이 없는 카메라는 `liveStreamName`, 그마저 없으면 `streamName`으로 폴백한다.
- 녹화 스트림 연결에 실패했을 때 별도 자동 재선택 로직은 추가하지 않는다. 기존 MSE 연결 상태 표시와 재연결 동작을 그대로 사용한다.

## 비목표

- 집중보기 전 고화질 스트림 미리 연결
- 일반 격자의 화질 변경
- 녹화 코덱, 해상도, 세그먼트 또는 ffmpeg 수명주기 변경
- 집중보기 전용 백엔드 API나 go2rtc 별칭 추가

## 검증

- 스트림 선택 규칙을 작은 자동 검사로 고정한다.
- 웹 lint와 production build를 실행한다.
- Go 전체 테스트와 daemon build를 실행한다.
- 실제 `/live`에서 일반보기는 라이브 스트림, 집중보기는 녹화 스트림 소비자로 연결되는지 확인한다.
- 집중보기 전후 recorder worker 입력과 프로세스가 바뀌지 않는지 확인한다.
