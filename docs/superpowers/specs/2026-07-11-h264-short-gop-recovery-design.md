# H.264 짧은 GOP 복구 설계

## 목적

4K HEVC 입력을 software H.264로 변환할 때, 입력 연결 직후 누락된 HEVC 참조 프레임이 첫 H.264 키프레임에 포함되어 화면 일부가 회색 또는 흰색으로 오래 남는 문제를 해결한다.

해상도 정책은 변경하지 않는다. 소방서5 집중보기는 계속 최대 1920×1080이며, 녹화는 3840×2160 HEVC copy를 유지한다.

## 확인된 원인

- 소방서5 원본 HEVC의 첫 일반 프레임은 `Could not find ref with POC` 오류와 함께 깨졌다.
- 다음 HEVC 키프레임은 3840×2160 전체가 정상 복원됐다.
- 현재 libx264 기본 설정은 12초 표본에서 H.264 키프레임을 1개만 생성했다.
- `-g 20 -keyint_min 20 -sc_threshold 0` 시험 설정은 6초 표본에서 키프레임 6개를 생성했다.

따라서 해상도가 아니라, 손상된 시작 프레임 이후 다음 H.264 IDR까지의 긴 대기 시간이 화면 깨짐을 지속시키는 원인이다.

## 변경 설계

go2rtc의 공용 software H.264 ffmpeg 템플릿에 다음 옵션을 추가한다.

```text
-g 20 -keyint_min 20 -sc_threshold 0
```

- 모든 software H.264 출력에 동일하게 적용한다.
- video copy 출력과 녹화 원본 copy에는 적용하지 않는다.
- 해상도, FPS, 오디오, activation, 카메라별 desired/applied 정책은 변경하지 않는다.
- 별도 설정 필드나 UI 제어는 추가하지 않는다. 복구 안정성을 위한 런타임 기본값으로 관리한다.

## 영향과 절충

- 10fps 입력은 약 2초마다, 20fps 입력은 약 1초마다 IDR을 생성한다.
- CPU 사용량 변화는 미미할 것으로 예상한다.
- 키프레임 증가로 H.264 출력 비트레이트는 소폭 증가할 수 있다.
- 짧은 GOP는 MSE 재접속과 패킷 손실 후 복구도 빠르게 한다.

## 테스트

1. 기존 정책 렌더링 테스트에 짧은 GOP 옵션이 없어서 실패하는 테스트를 먼저 추가한다.
2. H.264 템플릿에 옵션을 추가해 테스트를 통과시킨다.
3. `go test ./internal/stream`과 `go test ./...`를 실행한다.
4. daemon을 빌드하고 `scripts/camstationctl.sh restart`로만 재시작한다.
5. 소방서5 focus 출력에서 키프레임 간격과 정상 1920×1080 프레임을 확인한다.
6. 소방서5 recording 4K HEVC copy와 live 640×360 H.264 정책이 유지되는지 확인한다.
7. 소방서3·4를 포함한 기존 라이브 출력이 재연결되는지 확인한다.

## 완료 기준

- software H.264 출력이 최대 20프레임 간격으로 IDR을 생성한다.
- 소방서5 집중보기의 손상된 시작 화면이 다음 IDR에서 1~2초 내 복구된다.
- 소방서5 녹화·라이브·집중보기의 DB desired/applied 정책은 변경되지 않는다.
- 전체 테스트와 daemon health 검증이 통과한다.
